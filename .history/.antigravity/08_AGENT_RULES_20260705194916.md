# EcoRecover AI Agent Rules
Version: 2.0

Status: MANDATORY

Purpose

This document defines how AI coding agents must work when contributing to EcoRecover.

Every implementation decision must follow these rules.

If another document conflicts with this one regarding implementation behaviour, this document takes precedence.

------------------------------------------------------------------------------

# Primary Goal

Do not generate code.

Engineer the product.

Every implementation should move EcoRecover closer to a production-quality Android application.

Never optimize for speed over quality.

------------------------------------------------------------------------------

# Project Understanding

EcoRecover is NOT a demo.

EcoRecover is NOT a CRUD application.

EcoRecover is NOT an e-commerce application.

EcoRecover is an on-demand e-waste pickup platform.

The application combines

Blinkit

Instamart

Uber

Material3

while using an intelligent valuation engine.

Always understand WHY a feature exists before implementing HOW.

# Existing Code Rule

The current project already exists.

Never rebuild it.

Never regenerate architecture.

Never replace working modules.

Always extend existing code.

Prefer modification over replacement.

------------------------------------------------------------------------------

# Backend Rule

Business logic belongs only in backend.

Never move pricing logic into Android.

Never duplicate calculations.

Never calculate

Product value

Recovery percentage

Metal prices

Market trends

inside Android.

Android displays backend results only.

------------------------------------------------------------------------------

# Existing Engines

The following engines are permanent.

economic_fetcher.py

ewaste_lookup.py

cost_fetcher.py

metal_market_engine.py

Never replace them.

Never duplicate them.

Extend around them.

Future engines must orchestrate these.

------------------------------------------------------------------------------

# Architecture Rule

Respect MVVM.

Respect Repository Pattern.

Respect Compose.

Respect Material3.

Respect existing navigation.

Never introduce another architecture.

------------------------------------------------------------------------------

# UI Rule

Every screen has one purpose.

One primary CTA.

One ViewModel.

Minimal user effort.

Never overload screens.

Prefer cards over tables.

Prefer search over browsing.

# Compose Rule

Compose only.

Material3 only.

Stateless composables whenever possible.

State belongs in ViewModels.

Never place business logic inside composables.

------------------------------------------------------------------------------

# ViewModel Rule

One ViewModel per screen.

Expose immutable StateFlow.

Never expose MutableStateFlow.

Never access Retrofit directly.

Repositories only.
